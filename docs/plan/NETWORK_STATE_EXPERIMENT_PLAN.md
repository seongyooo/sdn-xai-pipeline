## 0. 이 문서의 범위

사용자가 제기한 질문은 사실 두 갈래다.

1. **Track A (진행 중)** — Exp-1 파싱 정확도 실험을 어떻게 마무리할지 (토폴로지/데이터셋/반복 횟수는 이미 [EVAL_PLAN.md](EVAL_PLAN.md)에서 결정됨 — 재확인만 필요)
2. **Track B (신규)** — "FlowRule을 고립된 상태로 적용하는 것"을 넘어, **실제 트래픽이 흐르고 혼잡도가 존재하는 네트워크에서 운영자가 상태를 모니터링하고 FlowRule을 추천받는** 시나리오. 이건 완전히 새로운 축이고, 이 문서의 핵심 내용이다.

D-32라는 제약 안에서 두 트랙을 어떻게 배분할지가 이 계획의 실질적인 결론이다.

## 1. 관련 연구 대비 포지셔닝

이 프로젝트가 직접 계승하는 원본 연구는 **NetIntent/IBNBench** (arXiv 2507.14398, 2025)다. 조사 결과 핵심 사실 세 가지:

| 항목 | NetIntent/IBNBench (원본) | 이 프로젝트 |
|---|---|---|
| 토폴로지 | 다이아몬드 4SW·4H (Mininet, ODL+ONOS) | **동일** (이미 일치) |
| 반복/통계 | 명시 없음 (재시도 횟수만 언급) | n=10 + bootstrap 95% CI — **더 엄격** |
| 배경 트래픽/혼잡 시뮬레이션 | **없음** — "ping으로 충분"하다고 명시, assurance는 packet/byte count 비교만 | **없음 (현재)** → 이 문서가 채우려는 공백 |
| 추천/재라우팅 | 없음 | 없음 (현재) → Track B |

**결론**: Track A는 이미 원본보다 통계적으로 엄격하게 설계되어 있으니 "왜 이렇게 설계했는가"를 논문에 정당화하는 것으로 충분하고, 새로 손댈 필요가 없다. Track B(혼잡 인지형 모니터링+추천)는 원본 연구에 아예 없는 공백이므로, **작게만 구현해도 논문의 독자적 기여(C6)로 쓸 수 있다.**

Track B의 현실적 구현 모델로 참고할 선례를 찾았다: **ONOS IMR(Intent Monitor and Reroute, Sanvito & Moro, NetSoft 2018)** — ONOS 컨트롤러에 실제로 내장된 서비스로, 인텐트별 트래픽 통계를 Traffic Matrix로 노출하고, 외부 애플리케이션이 이를 보고 재라우팅을 결정하는 구조다. 알고리즘은 **복잡한 ML이 아니라 단순 greedy** — 수요를 내림차순 정렬 후 capacitated residual graph에서 최단경로를 순차 선택한다. 이 패턴이 D-32 안에서 재현 가능한 유일하게 현실적인 접근이라고 판단한다. (Graph MPNN 기반 congestion-aware routing 같은 최신 ML 연구도 있지만, 한 달 내 재현은 비현실적 — "향후 연구" 인용 수준으로만 사용 권장)

## 2. Track A — 기존 Exp-1 설계 재확인 (변경 없음)

[EVAL_PLAN.md](EVAL_PLAN.md)에 이미 명시된 결정을 그대로 유지할 것을 권장한다. 근거:

- **토폴로지**: Small(4SW/4H) 기본 + Large(8SW/16H) 일반화 — NetIntent와 동일 규모라 직접 비교 가능. 변경 불필요.
- **데이터셋 (2026-07-23 업데이트)**: 60케이스 자체 데이터셋 대신, 팀원이 작성한 **GOLD-350**(350케이스, 7카테고리 균등, 독립 이중 라벨링+adjudication, κ=1.000)으로 교체 확정. `experiments/eval/convert_gold350.py`로 변환, Gold Validation도 이미 완료(accepted 300/300 Stage1→2 컴파일 PASS) — 상세는 [[EVAL_PLAN]] §15. **기존 T-D 진행분은 무효 — 처음부터 재실행 필요.**
- **반복 횟수**: n=10 + bootstrap CI(10,000 resamples) — 원본보다 엄격한 설계임을 논문에 명시할 것.
- **다음 작업**: T-D 나머지 rep 완료 → T-C, T-B, T-A 순서로 10 rep씩 실행 → `score_exp1.py --run-id <해당 run_id>`로 채점 (지난 세션에서 stray 로그 오염 방지 위해 `--run-id` 필수로 수정함, [[qa_2026_07_23_high_fixes]] 참고).

T-D 결과(2026-07-22)에서 이미 확인된 것: reroute 카테고리만 NEM 0.667로 약함(“reroute”라는 단어가 없는 경유 지정 intent를 LLM이 forward로 오분류). 이는 Track B 설계에 직접 영향을 준다 — 아래 4장 참조.

## 3. Track B — "실제 네트워크 상황" 구현 설계

### 3-1. 핵심 아키텍처 결정: 새 Stage가 아니라 "모니터링 루프 → 기존 파이프라인 재사용"

가장 중요한 설계 판단: 추천 기능을 만들 때 Stage 1(LLM 자연어 파싱)부터 새로 태우지 않는다.

```
[기존: 요청-응답형]
운영자 자연어 입력 → Stage1 LLM 파싱 → Stage2~6

[신규: 상시 모니터링형]
ONOS port_statistics 폴링(상시) → 혼잡 탐지(임계치) → greedy 재라우팅 후보 계산
                                                              │
                                                              ▼
                                          IntentIR(action="reroute", via_device=..., avoid_device=congested_sw)
                                          를 코드로 직접 생성 (Stage 1 LLM 건너뜀)
                                                              │
                                                              ▼
                                    Stage 2(컴파일) → Stage 3(정적검증) → Stage 4(Digital Twin) → Stage 5(XAI)
                                                              │
                                                              ▼
                                        운영자에게 "추천 카드"로 제시 → 승인 시에만 Stage 6 배포
```

**왜 이 구조인가:**
1. T-D 결과가 보여주듯 reroute는 LLM이 가장 취약한 카테고리다. 추천을 LLM 자연어 생성 경로로 만들면 그 취약점을 그대로 상속한다. 반대로 **코드가 결정론적으로 IntentIR을 생성**하면(C2 "결정론적 컴파일러"와 동일한 철학), 이 취약점을 원천적으로 피한다.
2. Stage 2~5는 이미 완성되어 있다. 재사용하면 정적 검증·Digital Twin 검증·XAI 설명이 추천에도 **공짜로** 적용된다 — "추천도 검증되고 설명된다"는 것 자체가 차별화 포인트가 된다.
3. 논문 서술이 자연스러워진다: "LLM은 의미 해석에만 쓰고 안전/최적화 판단에는 관여시키지 않는다"는 기존 논지(C1~C5)가 Track B에도 일관되게 이어진다.

### 3-2. 배경 트래픽 / 혼잡 시뮬레이션

**이미 있는 기반** (새로 만들 필요 없음):
- `pipeline/stage4_twin/topology.py`의 다이아몬드 토폴로지는 **이미 s1-s2-s4(1Mbps 저속) / s1-s3-s4(10Mbps 고속) 이중 경로**로 설계되어 있다. 지금까지는 이 대역폭 차이가 거의 활용되지 않았는데, 혼잡 시나리오 실험에는 정확히 필요한 구조다.
- `twin_verifier.py`에 `_iperf_check()`가 이미 구현되어 있다 (iperf3, 백업으로 iperf v2).
- `onos_client.py`에 `port_statistics()`가 이미 있다 (포트별 byte/packet count).

**추가로 필요한 것:**
- **배경 트래픽 생성기**: 검증 대상 flow와 무관한 호스트 쌍 사이에 지속적인 iperf3 트래픽을 흘려 특정 링크를 의도적으로 포화시키는 헬퍼. 예: h2→h3를 저속 경로(s1-s2-s4)로 강제해 1Mbps 링크를 90%+ 활용하게 만든 뒤, 그 상태에서 h1→h4 트래픽에 대한 추천이 어떻게 나오는지 관찰.
- **혼잡 지표 계산**: `port_statistics()`를 일정 간격(예: 1~2초)으로 폴링해 `utilization% = Δbytes / interval / link_capacity`를 계산하는 얇은 레이어. D-ITG 같은 별도 트래픽 생성기 도입은 시간 대비 이득이 낮음 — iperf3만으로 충분.

### 3-3. 지속 모니터링 (운영자 대시보드)

**이미 있는 기반**: `api.py`가 SSE로 파이프라인 진행 상황을 스트리밍하는 패턴이 이미 있고(`/api/run`), `static/app.js`에 `onTwinBw` 핸들러와 실시간 대역폭 배지 UI가 이미 존재한다(단, 현재는 Digital Twin 검증 실행 중에만 동작).

**확장 방향**: 이 이벤트 스트림을 "파이프라인 실행 중에만"이 아니라 **독립적인 상시 폴링 루프**로 분리한다. 신규 엔드포인트 `GET /api/monitor/stream` (SSE)이 `port_statistics()`를 주기적으로 폴링해 토폴로지 뷰의 링크 색상/굵기를 실시간 갱신. 기존 `app.js`의 배지 렌더링 로직을 재사용 가능.

### 3-4. 추천 알고리즘 (ONOS IMR 패턴)

1. 혼잡 탐지: 특정 링크 utilization이 임계치(예: 80%) 초과 지속(예: 5초) 시 트리거
2. 해당 링크를 지나는 활성 flow 중 재라우팅 대상 선정 (가장 큰 소비자 우선 — greedy)
3. 남은 capacity 그래프에서 대안 경로 계산 (다이아몬드 토폴로지에서는 사실상 "다른 경로 쓰기" 하나뿐이라 Dijkstra까지도 필요 없음 — 그래프가 커지면 필요)
4. `IntentIR(action="reroute", src_ip=.., dst_ip=.., via_device=<대안 경로 스위치>)` 생성 → 3-1의 파이프라인 통과 → XAI 설명 포함 추천 카드로 운영자에게 제시

### 3-5. 토폴로지 선택

**다이아몬드(4SW/4H) 재사용을 권장.** 이미 이중 경로 대역폭 차이가 설계돼 있고, Track A와 동일 토폴로지라 "같은 환경에서 파싱도 잘 되고 추천도 잘 된다"는 일관된 서술이 가능하다. Large(8SW/16H)로 확장하는 건 시간이 남을 때의 stretch goal로 미룬다 — 그래프가 커지면 진짜 최단경로 알고리즘이 필요해져 개발 비용이 뛴다.

### 3-6. 데이터셋 설계 — Exp-1과 근본적으로 다른 지점

Exp-1의 gold는 **사람이 수작업으로 작성**해야 했다(60케이스, 검증 미완료 상태). Track B는 그럴 필요가 없다는 게 핵심 이점이다:

- 시나리오 = "어떤 배경 트래픽을 얼마나 흘렸는가"라는 **설정값**이지, 사람이 라벨링할 자연어 텍스트가 아니다.
- 우리가 링크 용량을 다 알고 있으므로 (다이아몬드 토폴로지가 통제된 시뮬레이션이라), **최적 경로는 그래프 알고리즘으로 프로그램적으로 계산 가능** → gold를 사람이 만들 필요가 없다.
- 따라서 "데이터셋 설계"는 시나리오 설계로 대체된다: 예) (a) 저속경로 포화 후 고속경로로 신규 요청, (b) 양쪽 경로 모두 포화, (c) 혼잡 없음(false-positive 추천 방지 확인), (d) 혼잡 해소 후 원복 여부 등 5~8개 시나리오면 충분.

### 3-7. 반복 횟수

Exp-1의 n=10 컨벤션을 그대로 가져올 필요는 없다. iperf 측정은 자연 분산이 있지만, Digital Twin 1회 왕복(토폴로지 기동+트래픽 생성+측정+rollback)이 실측 기준 수십 초 걸리는 점을 고려하면 **시나리오당 n=3~5**를 권장한다(평균±표준편차 보고만으로 충분, bootstrap CI는 생략 가능한 수준의 표본).

### 3-8. 평가 지표 (신규 실험, 가칭 Exp-NET)

| 지표 | 정의 |
|---|---|
| `congestion_detection_accuracy` | 실제 혼잡 상황을 올바르게 탐지한 비율 (false positive/negative 포함) |
| `recommendation_optimality` | 추천 경로 vs 그래프 알고리즘 기준 최적 경로 일치율 |
| `post_recommendation_utilization` | 추천 적용 후 링크 utilization 변화(개선 폭) |
| `recommendation_to_approval_latency` | 혼잡 탐지 → XAI 설명 포함 추천 카드 생성까지 걸린 시간 |
| `explanation_grounding` | 추천 XAI 설명이 실제 measured 값(utilization 수치 등)을 인용하는지 (정성 체크) |

**운영자 사용자 스터디는 하지 않는다** — 조사 결과 SDN XAI 추천 분야에 표준화된 사용자 스터디 선례가 없고, D-32 내 실행이 비현실적이다. 논문에는 "정량 평가 중심, 사용자 스터디는 future work"로 명시한다.

## 4. 일정 및 우선순위 (D-32 기준)

| 우선순위 | 트랙 | 작업 | 예상 소요 |
|---|---|---|---|
| 🔴 필수 | A | T-D 나머지 rep + T-C/T-B/T-A 실행·채점 | ~1주 (API rate limit 지배적) |
| 🔴 필수 | A | Gold Validation 완료 (54케이스 컴파일 확인) | ~반나절 |
| 🔴 필수 | A | Exp-1 결과 분석 + 논문 5.1절 표 작성 | ~2~3일 |
| 🟡 권장(C6 후보) | B | 배경 트래픽 헬퍼 + 혼잡 지표 계산 레이어 | ~2~3일 |
| 🟡 권장(C6 후보) | B | Greedy 추천 알고리즘 + IntentIR 자동 생성 | ~2~3일 |
| 🟡 권장(C6 후보) | B | 상시 모니터링 SSE 엔드포인트 + 대시보드 확장 | ~2일 |
| 🟡 권장(C6 후보) | B | 시나리오 5~8개 × n=3~5 실행 + 표 작성 | ~2일 |
| 🟢 여유 있으면 | B | Large 토폴로지로 확장, 실제 최단경로 알고리즘 | 미정 |
| 🟢 여유 있으면 | — | 논문 작성/투고 준비 | 마감 직전 1주 확보 필요 |

**권장 판단**: Track A를 먼저 완주해 논문의 핵심 주장(C1~C5)에 대한 수치적 근거를 먼저 확보한다. Track B는 위 표의 4개 항목을 순서대로 진행하되, 논문 작성을 위해 **최소 1주는 마감 전 버퍼로 남겨야 한다.** Track B가 D-15 시점까지 끝나지 않으면 "Stage 7 프로토타입 + 5개 시나리오 케이스 스터디" 수준으로 축소하고, 정식 실험 결과 표 대신 정성적 예시(혼잡 전/후 스크린샷 + XAI 설명 1~2개)로 논문에 넣는 것으로 스코프를 낮춘다.

## 5. 사용자 결정이 필요한 열린 질문

- Track B를 논문의 **정식 6번째 기여(C6)**로 넣을지, 아니면 **"future work" 섹션의 proof-of-concept**로만 언급할지 — 이건 D-32 진행 상황을 보며 D-15 시점에 재판단하는 걸 권장.
- 혼잡 탐지 임계치(80%?), 지속 시간(5초?) 등 구체적 파라미터 — 3-4에 예시로 넣었지만 확정은 아님.
- 상시 모니터링을 실제 프로덕션 ONOS 대상으로 할지, Digital Twin(Mininet) 대상으로만 검증할지 — 후자가 훨씬 안전하고 통제 가능하므로 논문 실험 범위는 Twin 한정을 권장.

## 6. 리스크

- **시간 리스크가 가장 크다.** Track A도 아직 완주 전이고(T-A/B/C 미실행), Track B는 신규 개발이다. 4장의 우선순위 표를 넘지 않는 선에서 진행하고, 매주 진행 상황을 이 문서에 체크리스트로 반영할 것을 권장(EVAL_PLAN.md 10장 방식 참고).
- Digital Twin은 Linux+root+Mininet 환경에서만 동작한다(`_check_platform()`). 배경 트래픽까지 추가하면 twin 1회 실행 시간이 늘어나 반복 실험 시 rate-limit이 아니라 **wall-clock 시간 자체**가 병목이 될 수 있다 — 사전에 1회 실측해서 3-7의 반복 횟수를 조정할 것.
