## 4. 버그 수정: Digital Twin reroute/qos/sfc 검증 오류

### 증상
reroute 인텐트를 실행했을 때 Digital Twin 검증이 "reach" 여부를 잘못 판단했다.

### 원인 (2가지)

1. **`action` 탐지 오류**: `twin_verifier.py`가 flow rule 구조(OUTPUT 명령 유무)로 action을 역추론했는데, reroute가 forward와 동일한 구조라 구분 불가
2. **`expect_reach` 오류**: `expect_reach = (action == "forward")`로 하드코딩되어 reroute/qos/sfc는 항상 false로 처리됨

### 수정

**`pipeline/stage2_flowrule/compiler.py`** — 모든 액션 반환에 `intent_action` 키 추가
```python
# block
return {"intent_action": "block", "flows": [...]}
# forward
return {"intent_action": "forward", "flows": [...]}
# qos, reroute, sfc 동일
```

**`pipeline/stage4_twin/twin_verifier.py`**
```python
# 변경 전
has_output = any(i.get("type") == "OUTPUT" for i in instructions)
action = "forward" if has_output else "block"
expect_reach = (action == "forward")

# 변경 후
if "intent_action" in flowrule:
    action = flowrule["intent_action"]
else:
    ...  # fallback
expect_reach = (action != "block")  # block만 차단, 나머지는 모두 도달 기대
```

## 5. 주요 신기능: 복합 인텐트(Compound Intent) 지원

### 배경
기존 파이프라인은 하나의 자연어 인텐트 → 하나의 FlowRule만 처리했다. 실제 네트워크 운영에서는 "HTTP는 허용하고 SSH는 차단하라" 같이 여러 정책을 동시에 요청하는 경우가 많다.

### 아키텍처 변경

복합 인텐트는 후방 호환을 유지하면서 병렬 경로로 처리된다:

```
자연어 인텐트
    ↓ Stage 1 (LLM)
    ├── 단일 룰 → IntentPrediction(program=IntentIR)  [기존 경로]
    └── 복합 룰 → IntentPrediction(compound=CompoundIntentIR)  [신규 경로]
        ↓ Stage 2
    FlowRule { intent_action: "compound", sub_rules: [...], flows: [...] }
        ↓ Stage 3
    Intra-conflict detection (복합 룰 간 충돌)
        ↓ Stage 4
    Digital Twin (각 sub-rule별 검증)
```

### 파일별 변경 내용

#### `models/intent_ir.py`
- `IntentPrediction`에 `compound: Optional[CompoundIntentIR]` 필드 추가
- `CompoundIntentIR` 클래스 신규 추가:
  ```python
  class CompoundIntentIR(BaseModel):
      rules: list[IntentIR]
      description: str = ""
  ```

#### `pipeline/stage1_intent/intent_parser.py`
- `SYSTEM_PROMPT` 전면 재작성: `{"rules": [...], "description": "..."}` 통일 포맷으로 출력
- 복합 인텐트 예시 추가 (Allow HTTP + Block SSH 등)
- `parse()`: `rules` 배열 유무로 단일/복합 라우팅
- `_parse_single()`: 구형 단일 JSON 포맷 하위 호환 처리
- `_parse_rules()`: 각 rule에 토폴로지 그라운딩 적용, len≥2이면 `CompoundIntentIR` 반환

#### `pipeline/stage2_flowrule/compiler.py`
- `compile_compound(compound: CompoundIntentIR) -> dict` 함수 신규 추가:
  ```python
  {
    "intent_action": "compound",
    "description": "...",
    "sub_rules": [각 sub-rule 결과],
    "flows": [모든 flow 합산]
  }
  ```

#### `pipeline/stage3_static/static_validator.py`
- `_check_intra_conflicts(flows)` 함수 신규 추가:
  - 동일 디바이스의 같은 트래픽에 상반된 액션 감지 (Intra-Shadowing)
  - catch-all 룰이 특정 프로토콜 룰을 가리는 경우 감지
- `validate()` Step 3: `intent_action == "compound"`일 때 intra-conflict 검사 실행

#### `api.py`
- Stage 1: `prediction.compound` 분기 처리
- Stage 2: `compile_compound()` vs `compile_flowrule()` 분기

## 7. E1 실험 분석 (sdn_intent-framework)

### 데이터셋 구성

| 항목 | 내용 |
|---|---|
| 위치 | `sdn_intent-framework/experiments/e1/` |
| 케이스 수 | 100개 (N001~N050 upstream + 50개 project-authored) |
| 카테고리 | Forwarding 42 / Security 23 / QoS 23 / Compound 2 / Rejection 10 |
| Gold 상태 | Provisional (이중 annotation 미완료) |
| SFC/Reroute 확장 | 별도 100케이스 (`intents_sfc_reroute.jsonl`) |

### 4가지 실험 트리트먼트

| 트리트먼트 | 설명 |
|---|---|
| E1-A | LLM → ONOS FlowRule 직접 생성 (IR 없음) |
| E1-B | LLM → IR, zero-shot |
| E1-C | LLM → IR + few-shot examples |
| E1-D | LLM → IR + few-shot + topology grounding |

**실행 모델**: `qwen3:8b` (Ollama, temperature=0.2) × 5 repetition × 100 case = 2,000 calls

### 주요 결과 (e1_results.md)

| Metric | E1-A | E1-B | E1-C | **E1-D** |
|---|---:|---:|---:|---:|
| Schema validity | 0.432 | 0.830 | 0.930 | 0.910 |
| Exact match | 0.020 | 0.200 | 0.218 | **0.368** |
| Rule-count acc. | 0.013 | 0.778 | 0.844 | 0.822 |
| Hallucination rate | 0.036* | 0.159 | 0.241 | **0.021** |
| Rejection rate | 1.000* | 0.400 | 0.700 | 0.680 |

\* E1-A의 수치는 대량 과잉거절(over-rejection)의 부산물

**Upstream-only 통제 비교 (task-equivalence confound 제거, N=50)**

| Metric | E1-A | E1-B | E1-C | **E1-D** |
|---|---:|---:|---:|---:|
| Exact match | 0.000 | 0.000 | 0.000 | **0.280** |
| Hallucination | 0.050 | 0.289 | 0.486 | **0.042** |

### 핵심 발견

1. **IR 형식이 schema validity를 크게 높인다** — 43% → 83~93%
2. **Few-shot 단독은 hallucination을 오히려 악화시킨다** — 0.159 → 0.241 (B→C)
3. **Topology grounding이 hallucination을 실질적으로 억제한다** — 0.241 → 0.021 (C→D)
4. **Grounding 효과는 `device` 슬롯 하나에 집중된다** — device accuracy 0.481 → 0.924

### IR 스키마 차이

E1 framework와 현재 파이프라인의 IR 스키마가 다름:

| E1 framework | 파이프라인 `IntentIR` |
|---|---|
| `action: "deny"` | `action: "block"` |
| `action: "prioritize"` | `action: "qos"` |
| `selector.source.ip` | `src_ip` |
| `enforcement.device` | `device_hint` |
| `enforcement.egress_port` | `out_port` |
| `selector.source: null` 허용 | src_ip 미필수 케이스 다수 존재 |
